04 GitHub Repository Setup

📁 Repo Structure

Create one mono-repo called market-intelligence-engine. Put both frontend/ and backend/ in the same repo. Easier to coordinate, one README, one set of issues.

🌿 Branch Strategy

Use main for production-ready code. Use dev as the integration branch. Each person works on their own feature/ branch and submits PRs into dev.

🔐 Secrets & .env

Never commit .env files. Add .env, .env.local, *.env to .gitignore. Store secrets in GitHub → Settings → Secrets and Variables for CI/CD.

Step 1 — One person creates the repo and everyone else forks/clones

bash Initial setup (Person 2 does this first)

git add .

# Create dev branch
git checkout -b dev
git push origin dev

Step 2 — Everyone else clones and creates their feature branch

bash Each team member runs this

git clone https://github.com/YOUR-ORG/market-intelligence-engine.git

# Always branch from dev, not main

git pull origin dev

# Person 1:
git checkout -b feature/frontend

# Person 2:
git checkout -b feature/backend-api

# Person 3:
git checkout -b feature/scraper-diff

# Person 4:
git checkout -b feature/ai-agents

Step 3 — Daily workflow (each person does this)

bash Daily git workflow 
# At start of each work session — pull latest from dev
git fetch origin
git merge origin/dev   # merge any teammates' merged work into your branch

# Work, then commit regularly (small, specific commits)
git add backend/agents/scraper_agent.py
git commit -m "feat(scraper): add SHA-256 hash check to skip unchanged pages"

# Push your feature branch
git push origin feature/scraper-diff

# When your feature is done → open a Pull Request on GitHub
# PR: feature/scraper-diff → dev (NOT main)

When to open PRs and how to review:

  • Person 2 opens the first PR after scaffold is done (hours 0–4). Everyone reviews the schemas.py so all 4 agree on data structure before writing code.
  • Person 3 opens a PR at hour 4 with scraper_agent.py working. Person 4 reviews — they need to consume scraper output in their agents.
  • Person 4 opens a PR at hour 10 with analyst_agent.py + prompts.py. Person 2 reviews to see how to integrate into Celery task.
  • Person 1 opens a PR at hour 10 with components built on mock data. Person 2 reviews that the API shapes match what frontend expects.
  • At hour 18, all 4 PRs are merged to dev. Everyone pulls from dev. Integration sprint begins — bugs are fixed directly on feature branches or hotfix branches.
  • At hour 40+, one final PR merges dev → main. Tag it as a release: git tag v1.0.0-hackathon
